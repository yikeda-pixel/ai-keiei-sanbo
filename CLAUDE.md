## 開発ルール

### コーディング規約

- コメントは原則書かない。WHYが非自明な場合のみ1行で記述する
- 不要な抽象化・将来のための設計は行わない
- セキュリティ脆弱性（SQLインジェクション、XSS、コマンドインジェクション等）を導入しない

### Git・GitHub 運用ルール

**コードを変更するたびに、必ずGitHubへプッシュすること。**

具体的な手順：

1. 変更をステージングする
   ```
   git add <変更ファイル>
   ```

2. コミットメッセージはWHYを中心に簡潔に（1〜2文）
   ```
   git commit -m "コミットメッセージ"
   ```

3. 変更後は即座にプッシュする
   ```
   git push origin <ブランチ名>
   ```

**禁止事項：**

- `--no-verify` でフックをスキップしない
- `git push --force` を main/master ブランチに実行しない
- `.env` や認証情報を含むファイルをコミットしない

**ブランチ戦略：**

- `main` : 本番相当の安定ブランチ
- `feature/<機能名>` : 機能開発ブランチ
- `fix/<バグ名>` : バグ修正ブランチ
